# Corpus validation — v0.2.0 checks

Before releasing the three v0.2.0 checks (`hardport`, `maporder`, `eventuallyeffect`) we ran them
against the same four codebases as the [v0.1 corpus](2026-07-16-corpus-run.md) —
kubernetes/kubernetes, grafana/grafana, prometheus/prometheus, testcontainers-go — with the four
v0.1 checks disabled, and hand-triaged the findings. This document accounts for every finding: what
fired, what we classified as a genuine rule misclassification, and what was a correct-but-noisy hit
that motivated narrowing the rules before release.

## Reproduction details

- **Tool, first pass (pre-fix):** flakylint @ `e165acc`
- **Tool, re-run (post-fix):** flakylint @ `f56827d` (after `fix: limit eventuallyeffect to
  count-dependent side effects` and `fix: ignore per-iteration accumulators in maporder`)
- **Go toolchain:** go1.26.2
- **Target repos:** grafana/grafana @ `5342253d`, prometheus/prometheus @ `2cf3239`,
  kubernetes/kubernetes @ `bd1a1b89` (shallow, HEAD 2026-07-18), testcontainers-go (root module +
  85 `modules/*` + `examples/nginx`, HEAD 2026-07-18)
- **Command:** `flakylint -exitfatal=false -httptestclose=false -parallelglobal=false
  -sleepassert=false ./...` (leaves only the three new checks active). Per-repo scope matched the
  v0.1 run so numbers are comparable: grafana `./pkg/...` plus each of the 10 nested `go.work`
  modules; kubernetes `./pkg/... ./cmd/... ./plugin/... ./test/...` with `GOFLAGS=-mod=vendor`
  (`./staging/...` skipped, as in v0.1); prometheus and testcontainers-go full `./...` (azurite
  module unbuildable upstream, same pre-existing `go.sum` gap as v0.1).
- **Sampling:** grafana's `eventuallyeffect` produced 47 findings; a seeded random sample of 25
  (**seed 42**) was triaged. Every other check/repo cell was small enough to triage in full.

### The 3 false positives (rule misapplied its own semantics)

- **prometheus, `eventuallyeffect`, dead writes (2).** `cmd/prometheus/main_test.go:919` (`err`) and
  `tsdb/head_test.go:9788` (`b`): a plain `x = v` overwrite inside the callback where the variable
  is unconditionally reassigned before it is ever read. The write cannot depend on the poll count,
  so the diagnostic mischaracterized the code.
- **grafana, `maporder`, nested-range mis-attribution (1).**
  `pkg/services/ngalert/schedule/schedule_unit_test.go:650`: an `assert.Equal` inside a map-range
  whose accumulators are actually built by ranging over **slices** and are declared per-iteration
  (fresh each map key), so their order cannot depend on map iteration. `eachAccWrite` walked into
  the nested slice-range and attributed those appends to the enclosing map range.

### The grafana 24 — correct under the old rule, not a misclassification

Of grafana's 25 triaged `eventuallyeffect` findings, **24 were the guarded last-write-wins capture
idiom** (`x, err = Get(); return err == nil`, with `x` read after `Eventually`). Under the
then-current rule — which flagged *any* captured-variable write in a poll callback — these were
**correctly flagged**: the code genuinely writes captured state on every tick. They are *not*
counted as false positives, because the rule did exactly what it was documented to do.

They are counted as **idiom noise**: on real, healthy suites this pattern is deterministic and
ubiquitous, so flagging it would make the check chatty without pointing at a real flake. That signal
— 24 noisy-but-correct hits against 1 genuinely valuable one — is what motivated the narrowing. The
single valuable grafana hit was `pkg/infra/nats/integration_test.go:75` (a cross-poll settle
baseline mutated across polls and asserted later — a real premature-baseline flake); the narrowed
rule no longer reports it, and recovering that class of read-before-write cross-poll state is a
recorded backlog item.

## The narrowing

Both offending rules were tightened before release:

- **`eventuallyeffect`** now flags only **count-dependent** effects — increments (`x++`, `x--`),
  compound assignments (`x += …`), self-appends (`x = append(x, …)`), and sends on a captured
  channel. Plain overwrites (`x = v`, including multi-assign) are silent by design: they are
  last-write-wins, which is deterministic. This kills all 4 prometheus hits and silences the 24
  grafana idiom hits, at the documented cost of the one nats cross-poll case.
- **`maporder`** now requires the accumulator to be **declared outside** the map-range statement;
  an accumulator declared inside the range body is per-iteration-fresh and its internal order cannot
  depend on map iteration order. This kills the grafana nested-range FP.

## Re-run — compatibility check (tool @ `f56827d`)

With the narrowed rules, all three checks report **zero findings** across all four repos:

| Check | kubernetes | grafana | prometheus | testcontainers-go |
|---|:---:|:---:|:---:|:---:|
| `hardport` | 0 | 0 | 0 | 0 |
| `maporder` | 0 | 0 | 0 | 0 |
| `eventuallyeffect` | 0 | 0 | 0 | 0 |

**This is a negative / compatibility result, not a headline accuracy number.** Zero findings here
means the narrowed checks do not fire on four mature suites that already hold ports open on `:0`,
sort before asserting, and keep polling callbacks pure (or use the last-write-wins idiom the checks
now permit) — i.e. dropping v0.2.0 into these codebases adds no new diagnostics. It says nothing
about how often the checks catch the anti-patterns they target, because these suites contain almost
none of them.

Because a flat zero can equally mean "clean corpus" or "broken tool", each zero was backed by
**synthetic positive controls** and candidate-site audits:

- Every check was exercised on crafted `_test.go` inputs and fired exactly once each:
  `net.Listen("tcp", ":8080")` → `hardport`; a map-range `append` asserted with `reflect.DeepEqual`
  → `maporder`; a captured `done++` inside an `assert.Eventually` callback → `eventuallyeffect`.
  Each emitted exit code 3, confirming the binaries were live and the empty corpus results are true
  negatives.
- `hardport`: candidate call sites were grep-audited. kubernetes has zero hardcoded test binds;
  prometheus routes every bind through `testutil.RandomUnprivilegedPort(t)`; grafana and
  testcontainers-go bind `…:0` (the exact pattern `hardport` recommends).
- `eventuallyeffect`: kubernetes standardizes on `wait.Poll*` (only 3 testify `Eventually` sites,
  all pure); testcontainers-go's single `Eventually` callback is a read-only predicate.
- `maporder`: real map ranges in the corpus either write back into the map (order-insensitive),
  assert per-key, or sort before comparing.

## Documented limits (unchanged by the narrowing)

- `hardport` only sees **constant** addresses; a port routed through a variable is silent (e.g.
  grafana `graphite_test.go:480` `port := ":56789"`, prometheus label/config literals).
- `eventuallyeffect` does not track **pointer-mediated** mutations (`json.Unmarshal(b, &state)` —
  the load-bearing value in prometheus `tsdb/head_test.go`), and after the narrowing no longer flags
  plain overwrites even when they feed a read-before-write cross-poll state machine (backlog).
- `maporder` only analyzes `TestXxx`/subtest bodies, so an order-dependency in a production helper
  that happens to live in a `_test.go` (e.g. testcontainers-go `modules/rabbitmq/types_test.go`) is
  out of scope by design.
